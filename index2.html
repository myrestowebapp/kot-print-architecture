Restaurant POS · Print Infrastructure

Many captains. Many kitchens. Many KOTs — zero collisions.

Bluetooth thermal printers accept exactly one connection at a time — and a busy restaurant rarely stops at one printer. Captains fire orders from four phones; one table can spawn a kitchen KOT, a bar KOT, and a follow-up add-on KOT minutes later. This architecture handles all of that through a Print Agent and dispatcher: one negotiated Bluetooth channel per printer, unlimited captain devices, automatic station routing, and the exact ticket format your desktop POS already produces.

Why this happens: Bluetooth was never built for this

01 / Problem

The thermal printers restaurants use talk over Bluetooth's Serial Port Profile (SPP) — the same protocol that made old wireless serial cables possible. SPP was designed as a point-to-point link: one paired device, one open channel. That's a firmware-level limit, not a configuration setting, so no app update on the printer side changes it.

In practice that means the moment Captain 1's phone holds the Bluetooth session, Captains 2 through 4 simply can't reach the printer. Their KOTs either fail silently, queue on-device with no way to flush, or force a manual re-pair dance mid-service — the worst possible time for it to break.

The fix isn't a better Bluetooth trick. It's making sure only one thing ever holds that Bluetooth session, and giving every other device a different, shareable way to reach it.

Root cause
Bluetooth SPP allows exactly one active connection per printer, by design.
Wrong fix
Trying to rotate Bluetooth pairing between 4 phones during live service.
Right fix
One device owns the Bluetooth link. Everyone else reaches it over a network.

The solution: a Print Agent between your captains and the printer

02 / Solution

One device — a spare captain phone or a small dedicated Android box per kitchen — keeps the Bluetooth pairing and runs a lightweight Print Agent. Every other device sends its KOT as a message, not a print job. In a multi-kitchen branch, each station gets its own agent; the dispatcher routes jobs by printer ID. Two ways to wire that message up — pick based on how much you want to depend on the restaurant's own WiFi.

Agent runs a local WebSocket server on the restaurant's own WiFi/LAN.

Inside the agent: why a queue, not just a relay

03 / Queue mechanics

Relaying messages to the printer isn't enough on its own. If two captains fire a KOT in the same second, two raw ESC/POS streams can interleave on the Bluetooth channel and print garbled tickets. The agent needs a queue in front of each printer, not just a pipe — and in a multi-kitchen branch, every station runs its own independent FIFO.

Receive
Job arrives with restaurant ID, target printer ID, KOT type, and order payload.
Route
Dispatcher resolves the printer ID and enqueues to that station's agent — not a global queue.
Enqueue
Job is appended to the agent's in-memory FIFO, tagged with a unique job ID and parent order ID.
Print, one at a time
Each agent drains its own queue sequentially — kitchen and bar can print in parallel.
Acknowledge
Success or failure is sent back to the originating captain and logged centrally.
Retry
Failed jobs (printer offline, out of paper) requeue with backoff instead of silently dropping.

Multi-KOT: one order, many tickets over time

04 / Multi-KOT

A table order is not a single print event. During a service, captains send new items, add-ons, cancellations, and course changes — each producing a separate KOT with its own ID, all linked back to the same parent order. The architecture treats every KOT as an independent print job while preserving order context for the kitchen.

NEW

Initial KOT

First fire for a table — all items grouped by station. One order can split into a kitchen ticket and a bar ticket in the same second.

ADD

Add-on / follow-up KOT

Guest orders two more dishes mid-meal. Only the new items print, clearly marked as an add-on with the original table and order ID so the kitchen knows it's not a duplicate.

MOD

Modification KOT

Quantity change, spice level, or item swap. Prints only the delta — what changed and what it changed to — so the station doesn't re-read the full order.

VOID

Cancel / void KOT

Item removed before it's made. Routes to the same station printer that received the original item, with a prominent void header so staff stop prep immediately.

RPT

Reprint KOT

Manager-triggered reprint of any prior KOT by job ID — same payload, new print timestamp. Useful when a ticket was lost or the printer jammed mid-service.

{
  "kot_id": "KOT-1042-003",
  "order_id": "ORD-1042",
  "kot_type": "ADD",
  "table": "T-12",
  "station": "kitchen",
  "printer_id": "printer_kitchen_01",
  "items": [{ "name": "Butter Naan", "qty": 2 }],
  "parent_kot_id": "KOT-1042-001"
}
Key rule: the captain app always sends one canonical KOT JSON per print event. The backend assigns kot_id, resolves printer_id from the item's station, and the dispatcher creates one job per KOT — never one job per order.

Four ways to move a KOT from phone to printer

05 / Transport options

The Print Agent pattern already fixes the Bluetooth ceiling. The remaining choice is which transport carries the job from captain to agent, and whether Bluetooth stays in the picture at all. Rated for fit against your goals: real-time, offline-tolerant, scalable past one location.

Recommended
1 · Print Server + WebSocket★★★★★
Captain 1‑4 ─┐
        ├─ WS Server ─ Print Agent ─ BT Printer
Captain N ──┘
  • One Bluetooth connection, unlimited captain devices
  • Sub-second latency, push-based — no polling delay
  • Native support for print queue, retries, reprints
  • Reuses your existing paired printer hardware as-is
2 · Network (WiFi/Ethernet) Printer★★★★★
Captain 1‑4 ─── ESC/POS over TCP:9100 ─── WiFi Printer
  • No Bluetooth limitation at all — printer accepts many sources
  • Industry standard for commercial multi-terminal POS
  • Still needs a dispatcher/queue in front to avoid interleaved streams
  • Requires new printer hardware — best for new installs
3 · Print Queue API (HTTP polling)★★★★★
Captain ─ HTTP API ─ DB Queue ─ Print Service (polls/sec) ─ Printer
  • Simplest to build — no persistent socket to manage
  • Noticeably slower: bound by poll interval, not event-driven
  • Fine as a fallback path when WebSocket is unavailable
4 · MQTT★★★★★
Captain ─ MQTT Broker ─ Print Service ─ Printer
  • Built for exactly this: many publishers, reliable delivery, auto-reconnect
  • Strong fit once you have several kitchen/bar/bakery printers per branch
  • Adds a broker to operate — more infrastructure than WebSocket for a single printer

What restaurant software already does about this

06 / Market patterns

None of this is a novel problem — it's the standard bottleneck every multi-terminal POS hits. Three patterns dominate the market, and they aren't mutually exclusive.

Most common

Print Server / Agent

A background service on one machine bridges every POS terminal to the physical printers. This is the pattern behind Toast, Petpooja, and Posist. It's exactly the Print Agent described above.

Fit: direct match for your setup
Hardware-native

Star CloudPRNT / Epson ePOS-Print

Some printer manufacturers bake a cloud endpoint into the printer's own firmware — the POS pushes a job to a URL and the printer polls or pulls it directly.

Fit: elegant, but locks you to specific printer brands
Paperless alternative

Kitchen Display System (KDS)

Orders route to a screen in the kitchen instead of a printer, removing the bottleneck entirely. Worth a look as a future option — though most kitchens here still expect a physical KOT.

Fit: longer-term, parallel offering

Multi-kitchen routing: every station, the right printer

07 / Multi-kitchen & routing

Once every client sends jobs through a dispatcher instead of printing directly, routing to the correct kitchen becomes a lookup, not a rewrite. Each printer registers with a unique ID, station type, and profile; each menu item maps to one or more stations. A branch with a main kitchen, tandoor, Chinese section, bar, and bakery runs one Print Agent per Bluetooth printer — each agent owns a single pairing and drains only jobs addressed to its printer ID.

WiFi/Ethernet printers (TCP 9100) don't have the Bluetooth limit. A single agent can hold connections to several network printers and route by printer ID.

Each menu item carries a station_id in the catalog. The dispatcher resolves station → printer ID at job creation. Override per-item for specials or pop-up stations.

Keeping desktop and mobile tickets identical

08 / One format, two agents

You already print from the desktop web POS through QZ Tray. Adding an Android Print Agent for captain phones creates a second code path — and a second place a KOT format can quietly drift out of sync with the first.

The fix: don't format ESC/POS on the client at all. Your backend generates one canonical KOT payload — table, order ID, items, quantities, notes, timestamp — and both QZ Tray and the Android Print Agent render it through the same shared Renderer Library, a single piece of logic that turns KOT JSON into ESC/POS commands and consults the printer registry for width and encoding. Change the renderer once, and every agent — desktop or mobile, today or future — updates together. A captain's ticket looks identical whether it printed from a phone or a desktop terminal.

  • 02 Foundation

    Define the canonical KOT schema

  • 03 Reliability

    Add ack, retry, and offline queueing

    Pilot in cloud-relay mode
